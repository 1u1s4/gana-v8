gana-v8 — diagrama técnico del runtime real
Monorepo pnpm + Turbo + TypeScript + Prisma/MySQL. El estado actual del código es un control plane Hermes-native que coordina workers especializados sobre un operational store MySQL/Prisma. public-api y operator-console son adapters de lectura/ops; no existe frontend web React/Next en producción dentro del repo

Plano de ejecución

  • El control plane persiste y reclama Task/TaskRun en MySQL; hoy esa tabla funciona como task queue lógica
  • ingestion-worker es el único que toca upstream real de data market/API en el repo: API-Football
  • research-worker, scoring-worker, publisher-worker y validation-worker son apps separadas y deberían pensarse como runtimes distintos
  • public-api compone read models desde Prisma; operator-console consume esos read models y no habla directo con DB

Persistencia exacta

  • Schema central: Fixture, RawIngestionBatch, OddsSnapshot, OddsSelectionSnapshot, Task, TaskRun, AiRun, Prediction, Parlay, Validation, AuditEvent
  • AiRun hoy representa scoring interno determinístico, no un proveedor LLM externo
  • source-connectors y canonical-pipeline viven como packages, pero el operational truth termina en MySQL/Prisma
  • storage-adapters implementa repos in-memory y Prisma con unit-of-work compartido entre apps

Notas técnicas clave

  • No dibujé Redis/Kafka/SQS porque no aparecen como infra real activa en el repo
  • No dibujé web frontend React/Next porque el repo actual no lo implementa
  • El sandbox existe como infraestructura sintética y tooling seguro, no como parte del runtime productivo principal
  • La arquitectura deseada habla de fronteras limpias por package, aunque todavía hay app→app adapters en el estado actual
