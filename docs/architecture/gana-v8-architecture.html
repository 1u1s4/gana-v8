gana-v8 — infraestructura real del monorepo
Estado actualizado desde el código: control plane + workers especializados + Prisma/MySQL como task queue y operational store + API-Football como integración upstream + public-api/operator-console como capa de observabilidad, no como web app tradicional

Topología real hoy

  • Monorepo pnpm + Turbo + TypeScript + Prisma/MySQL
  • La arquitectura está dominada por control plane y workers, no por una web app tradicional
  • public-api y operator-console existen como capa de lectura/observabilidad
  • operator-console hoy es un renderer textual, no una consola React/Next productiva

Loop operativo implementado

  • API-Football → ingestion-worker → raw batches / odds snapshots / fixtures
  • control plane usa Task + TaskRun en MySQL como task queue lógica
  • research-worker enriquece fixture metadata y scoring-worker persiste AiRun + Prediction
  • publisher-worker arma Parlay y validation-worker liquida artifacts para replay/health

Qué corregí respecto al diagrama viejo

  • Separé research, scoring, publisher y validation como workers reales distintos
  • Reemplacé la idea vaga de CLI/Telegram por sandbox-runner y operator-console reales
  • Hice explícito que la DB es MySQL/Prisma y que no hay Redis/Kafka/SQS productivos hoy
  • El modelo de AI actual es interno/determinístico; no dibujé un proveedor LLM externo inexistente
